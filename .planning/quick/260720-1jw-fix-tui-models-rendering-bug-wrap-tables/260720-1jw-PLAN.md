# Quick Task 260720-1jw: Fix TUI `/models` rendering bug

## Objective

The TUI `/models` command prints `(<rich.table.Table object at 0x...>, <rich.table.Table object at 0x...>)`
instead of rendering the two tables. Root cause: `_models_tables(cfg)` in
`src/agent86/tui/commands.py` returns a **tuple** `(table, roles)`, but `CommandResult.render`
is contracted to be a single str / Rich renderable. `app.py` writes it via
`RichLog.write(result.render)` (app.py:135), and `RichLog.write` stringifies a non-renderable
tuple into its `repr`. `/model` (singular, no arg) returns a plain string and is unaffected.

## Fix

Wrap the two tables in a single `rich.console.Group` so `_models_tables` returns one renderable.
`RichLog.write` renders a `Group` correctly. Update the pinning test that currently asserts the
buggy tuple return.

## Tasks

### Task 1: Return a single renderable from `_models_tables`
- **files**: `src/agent86/tui/commands.py`
- **action**: In `_models_tables(cfg)`, import `Group` from `rich.console` (lazily, alongside the
  existing lazy `Table` import) and return `Group(table, roles)` instead of `table, roles`.
- **verify**: `handle_command(repl, "/models").render` is a single Rich renderable (a `Group`),
  not a tuple.
- **done**: `/models` renders both the Providers and Model-roles tables in the TUI transcript.

### Task 2: Update the pinning test
- **files**: `tests/tui/test_commands.py`
- **action**: In `test_models_vs_model_routing`, replace the tuple assertions
  (`isinstance(result.render, tuple)` / `len(result.render) == 2`) with an assertion that

  `__rich_console__` / is renderable) and is NOT a tuple. Keep the `/model` string assertions
  unchanged.
- **verify**: `python -m pytest tests/tui/test_commands.py -q` passes.
- **done**: Test pins the corrected single-renderable contract; full TUI suite green.
